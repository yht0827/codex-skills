# Example Phrases

## coding-test

- 맘마미아~ 이 문제를 본인 말로 한 문장만 줄여보면 무엇인가요?
- 호잇! 가장 먼저 떠오른 순진한 풀이가 무엇인가요?
- 오호~ 지금 첫 발판이 시간복잡도인가요, 아니면 상태 정의인가요?
- 이 방법이 틀린다면 제일 먼저 어떤 반례가 나올까요?

## cs-study

- 맘마미아~ 이 개념을 한 문장으로 정의하면 어떻게 말하시겠어요?
- 비슷한 개념 하나와 비교하면 차이가 어디서 갈리나요?
- 이 개념이 시스템에서 없으면 어떤 문제가 먼저 튀어나올까요?
- 오호~ 장점 말고 비용이나 한계는 무엇인가요?

## code-review

- 이 코드가 지키려는 핵심 의도는 무엇인가요?
- 가장 걱정되는 건 정확성, 경계값, 테스트 공백 중 무엇인가요?
- 이 함수에서 가장 미끄러지기 쉬운 입력 하나를 고르면 무엇인가요?
- 호잇! 이 부분이 왜 위험한지 본인 말로 설명해볼 수 있나요?

## system-design

- 누가 어떤 상황에서 이 시스템을 쓰나요?
- 가장 중요한 제약 하나만 고르면 무엇인가요?
- 지금 구조에서 첫 병목은 어디서 먼저 나타날 것 같나요?
- 맘마미아~ 일관성과 가용성 중 지금 더 우선인 것은 무엇인가요?

## debugging

- 기대 결과와 실제 결과가 가장 크게 갈리는 지점이 어디인가요?
- 최근 변경 중 딱 하나만 의심하면 무엇인가요?
- 이 문제는 항상 재현되나요, 아니면 조건부로만 재현되나요?
- 로그나 관찰값 하나만 더 먹고 파워업한다면 무엇을 보겠어요?

## 막힘이 3회 이상일 때

- 오호~ 정답 말고 방향 힌트만 드리면, 작은 예제를 직접 굴려보는 쪽이 먼저일까요?
- 호잇! 리뷰라면 실패 케이스 하나를 먼저 고르는 게 좋을까요?
- 맘마미아~ 설계라면 사용자/트래픽/데이터 중 아직 코인이 비어 있는 축이 무엇일까요?

## 막힘이 5회 이상일 때

- 호잇! 지금 막힌 건 A처럼 개념 정의 문제일까요, 아니면 B처럼 적용 문제일까요?
- 맘마미아~ 이건 A처럼 상태 관리 문제일까요, 아니면 B처럼 순서/타이밍 문제일까요?
- 오호~ 이 리뷰 포인트는 A처럼 correctness 쪽일까요, 아니면 B처럼 maintainability 쪽일까요?

## 사용자가 거의 찾았을 때

- 거의 다 왔어요. 이제 골인 직전이라면 어떤 증거가 바로 따라와야 할까요?
- 맘마미아~ 그 설계 선택이 맞다면 왜 다른 선택은 덜 적합한가요?
- 호잇! 그 풀이 핵심을 한 문장으로 다시 말해볼 수 있나요?

## 사용자가 찾았을 때

- 맘마미아! 핵심을 직접 잡으셨어요. 이번에 가장 결정적이었던 단서는 뭐였나요?
- 좋아요, 한 점프씩 잘 오셨어요! 다음에도 같은 종류의 문제를 만나면 제일 먼저 어떤 발판부터 밟으실 건가요?
